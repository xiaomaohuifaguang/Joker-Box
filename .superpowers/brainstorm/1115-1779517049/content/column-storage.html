动态表格列定义存储方案

列定义是动态表格的核心配置，选择存储位置决定了数据模型和校验方式

A

复用 options 字段（推荐）

列定义存入 DynamicFormField.options，每个 DynamicFormOption 表示一列：

// field.options = [
//   { label: "姓名", value: "name" },
//   { label: "年龄", value: "age" },
//   { label: "电话", value: "phone" }
// ]
//
// 优点：
// ✓ 无需新增数据库字段
// ✓ label/value 天然对应 列标题/列key
// ✓ 前端已有 options 渲染逻辑，适配成本低
// ✓ 发布校验可复用 validateOptions 逻辑
//
// 缺点：
// ✗ value 字段语义从"选项值"变为"列key"，略不直观
// ✗ children 字段无意义（TABLE不用嵌套）
B

复用 props 字段

列定义存入 DynamicFormField.props 的 columns 属性：

// field.props = {
//   columns: [
//     { key: "name",  title: "姓名" },
//     { key: "age",   title: "年龄" },
//     { key: "phone", title: "电话" }
//   ]
// }
//
// 优点：
// ✓ key/title 语义更准确
// ✓ props 本就是扩展配置位，符合设计意图
// ✓ 不影响 options 的原有语义
//
// 缺点：
// ✗ 校验逻辑需单独编写（不能复用 validateOptions）
// ✗ columns 与 options 并存时前端需区分处理
// ✗ 列定义和选项列表是两套数据结构
C

新建专用字段 columns

在 DynamicFormField 和数据库表中新增 columns JSON 字段：

// 新增字段: cat_dynamic_form_field.columns (JSON)
// field.columns = [
//   { key: "name",  title: "姓名" },
//   { key: "age",   title: "年龄" },
//   { key: "phone", title: "电话" }
// ]
//
// 优点：
// ✓ 语义最清晰，专列专用
// ✓ 校验独立，不影响其他类型
//
// 缺点：
// ✗ 需要数据库 DDL 变更
// ✗ 需要新增 TypeHandler 和校验逻辑
// ✗ 仅 TABLE 类型使用，字段利用率低
